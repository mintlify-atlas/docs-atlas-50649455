## Overview

Memory planning computes a static allocation scheme for activation tensors, using lifetime analysis to reuse buffers and minimize peak memory consumption. This happens after kernel matching and before code emission.

## Memory Plan Structure

The memory planner produces a `MemoryPlan` containing:

```rust
pub struct MemoryPlan {
    pub activation_buffers: Vec<BufferAllocation>,
    pub kv_cache: KVCacheLayout,
    pub total_activation_bytes: usize,
    pub total_kv_cache_bytes: usize,
    pub total_weight_bytes: usize,
    pub decode_regions: Vec<DecodeRegion>,
    pub decode_tensor_region: HashMap<TensorId, u32>,
}
```

This separates memory into three categories:
- **Activations**: Temporary tensors computed during forward pass
- **KV Cache**: Persistent attention cache across tokens
- **Weights**: Model parameters (memory-mapped from disk)

## Lifetime-Based Aliasing

The core algorithm uses greedy interval-graph coloring to reuse buffers whose lifetimes don't overlap.

### Liveness Analysis

From `src/compile/memory.rs`:

```rust
fn compute_liveness(graph: &CompGraph) -> HashMap<TensorId, BufferLifetime> {
    let mut liveness: HashMap<TensorId, BufferLifetime> = HashMap::new();

    for (node_idx, node) in graph.nodes.iter().enumerate() {
        // The output tensor is first written here
        let tid = node.output;
        if let Some(t) = graph.tensors.get(&tid) {
            if matches!(t.storage, StorageClass::Activation { .. }) {
                let entry = liveness.entry(tid).or_insert(BufferLifetime {
                    first_write: node_idx,
                    last_read: node_idx,
                });
                entry.first_write = entry.first_write.min(node_idx);
            }
        }

        // Input tensors are last read at this node
        for input_tid in &node.inputs {
            if let Some(t) = graph.tensors.get(input_tid) {
                if matches!(t.storage, StorageClass::Activation { .. }) {
                    let entry = liveness.entry(*input_tid).or_insert(BufferLifetime {
                        first_write: node_idx,
                        last_read: node_idx,
                    });
                    entry.last_read = entry.last_read.max(node_idx);
                }
            }
        }
    }

    liveness
}
```

This computes for each activation tensor:
- `first_write`: Node index where the tensor is produced
- `last_read`: Node index where the tensor is last consumed

### Overlap Detection

```rust
pub struct BufferLifetime {
    pub first_write: usize,
    pub last_read: usize,
}

impl BufferLifetime {
    pub fn overlaps(&self, other: &BufferLifetime) -> bool {
        self.first_write <= other.last_read && 
        other.first_write <= self.last_read
    }
}
```

Two buffers can be aliased if their lifetimes don't overlap - that is, one is completely finished before the other begins.

## Greedy Allocation Algorithm

The planner sorts activations by size (largest first) for better packing:

```rust
pub fn plan_memory(graph: &CompGraph) -> MemoryPlan {
    let liveness = compute_liveness(graph);

    // Sort activations by size descending
    let mut activations: Vec<(TensorId, usize, BufferLifetime)> = liveness
        .into_iter()
        .map(|(tid, lt)| {
            let t = graph.tensor(tid);
            let size = t.max_size_bytes();
            (tid, size, lt)
        })
        .collect();
    activations.sort_by(|a, b| b.1.cmp(&a.1));

    // Greedy interval coloring
    let mut buffers: Vec<BufferAllocation> = Vec::new();
    let mut tensor_to_buffer: HashMap<TensorId, BufferId> = HashMap::new();

    for (tid, size, lt) in &activations {
        // Try to reuse existing buffer
        let reused = buffers.iter_mut().find(|b| {
            !b.lifetime.overlaps(lt) && b.size_bytes >= *size
        });

        if let Some(buf) = reused {
            // Reuse - extend lifetime to cover both uses
            buf.lifetime.first_write = buf.lifetime.first_write.min(lt.first_write);
            buf.lifetime.last_read = buf.lifetime.last_read.max(lt.last_read);
            buf.aliases.push(*tid);
            tensor_to_buffer.insert(*tid, buf.id);
        } else {
            // Allocate new buffer
            let id = BufferId(buffers.len() as u32);
            let offset = buffers.iter()
                .map(|b| b.offset + b.size_bytes)
                .max()
                .unwrap_or(0);
            // Align to 16 bytes
            let offset = (offset + 15) & !15;
            buffers.push(BufferAllocation {
                id,
                size_bytes: *size,
                offset,
                aliases: vec![*tid],
                lifetime: *lt,
            });
            tensor_to_buffer.insert(*tid, id);
        }
    }
}
```

<Note>
The greedy approach is not optimal but runs in O(n²) time and produces good results in practice. Sorting by size helps pack large tensors first.
</Note>

## Buffer Allocation

Each buffer allocation tracks:

```rust
pub struct BufferAllocation {
    pub id: BufferId,
    pub size_bytes: usize,
    pub offset: usize,
    pub aliases: Vec<TensorId>,  // Tensors sharing this buffer
    pub lifetime: BufferLifetime,
}
```

The `aliases` vector shows which tensors reuse the same physical buffer at different points in execution.

### Alignment

All buffers are aligned to 16-byte boundaries for Metal compatibility:

```rust
let offset = (offset + 15) & !15;
```

## Decode Regions

For concurrent dispatch in decode mode, each buffer becomes a separate Metal buffer:

```rust
// Build decode regions from buffer allocations
let mut decode_regions = Vec::with_capacity(buffers.len());
let mut decode_tensor_region: HashMap<TensorId, u32> = HashMap::new();

for buf in &buffers {
    let region_id = buf.id.0;
    let mut max_decode_bytes = 0usize;
    
    for &tid in &buf.aliases {
        let t = graph.tensor(tid);
        let decode_bytes = t.decode_size_bytes();
        max_decode_bytes = max_decode_bytes.max(decode_bytes);
        decode_tensor_region.insert(tid, region_id);
    }
    
    // Align to 16 bytes
    max_decode_bytes = (max_decode_bytes + 15) & !15;
    
    decode_regions.push(DecodeRegion {
        region_id,
        size_bytes: max_decode_bytes,
        tensors: buf.aliases.clone(),
    });
}
```

<Info>
Decode regions enable fine-grained memory barriers. Since each region is a separate MTLBuffer, barriers can synchronize only the specific buffers with read-after-write hazards.
</Info>

## KV Cache Layout

The KV cache stores attention key and value vectors for all previous tokens:

```rust
fn plan_kv_cache(graph: &CompGraph) -> KVCacheLayout {
    let p = &graph.metadata.params;
    let num_layers = p.num_hidden_layers;
    let kv_heads = p.num_kv_heads;
    let head_dim = p.head_dim;
    let max_seq = p.max_position_embeddings;
    let dtype = DType::F16;
    let bytes_per_elem = 2usize;

    let per_layer_key_bytes = kv_heads * max_seq * head_dim * bytes_per_elem;
    let per_layer_val_bytes = kv_heads * max_seq * head_dim * bytes_per_elem;

    let mut layers = Vec::with_capacity(num_layers);
    let mut offset = 0usize;

    for layer_idx in 0..num_layers {
        let key_offset = offset;
        offset += per_layer_key_bytes;
        let value_offset = offset;
        offset += per_layer_val_bytes;
        
        layers.push(KVCacheLayerLayout {
            layer_idx,
            key_offset,
            key_size_bytes: per_layer_key_bytes,
            value_offset,
            value_size_bytes: per_layer_val_bytes,
            dtype,
        });
    }

    KVCacheLayout {
        total_bytes: offset,
        layers,
    }
}
```

For Llama 3.2 3B:
- 28 layers × 8 KV heads × 128 head_dim × 8192 max_seq × 2 bytes × 2 (key+value)
- ≈ 896 MB for full KV cache

<Warning>
The KV cache is allocated for maximum sequence length upfront. For long-context models, this can be the dominant memory consumer.
</Warning>

## Weight Memory

Weights are memory-mapped from safetensors files:

```rust
fn compute_weight_bytes(graph: &CompGraph) -> usize {
    graph.tensors.values()
        .filter(|t| matches!(t.storage, StorageClass::Weight(ref w) if w.byte_size > 0))
        .map(|t| {
            if let StorageClass::Weight(ref w) = t.storage {
                w.byte_size
            } else {
                0
            }
        })
        .sum()
}
```

The weight binding resolution pass (run before memory planning) has already filled in byte offsets for zero-copy access.

## Optimization Impact

Lifetime-based aliasing typically achieves:
- **2-4× reduction** in activation memory vs. naive allocation
- **Constant-time lookups** for tensor-to-buffer mapping
- **Zero runtime overhead** - all decisions made at compile time

<Tip>
For debugging memory issues, the compiler can emit a visualization of buffer lifetimes showing which tensors alias to which buffers.
</Tip>
